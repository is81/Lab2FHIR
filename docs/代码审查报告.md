## 一、严重问题 (Critical)

### 1.1 硬编码的绝对路径导致数据库不可移植

- **文件**：`backend/src/db/models.py` 第5行
- **严重程度**：Critical
- **问题**：`DATABASE_URL` 默认值硬编码为 `"sqlite:///f:/Lab2FHIR/backend/lab2fhir.db"`，这是一个 Windows 绝对路径。在 Linux/Mac 或他人机器上运行时，这个路径根本不存在，会导致应用无法启动。
- **修复建议**：使用相对路径 `"sqlite:///./lab2fhir.db"` 或使用 `os.path` 基于当前文件位置动态计算路径。

```python
# 建议改为
import os
_BASE_DIR = os.path.dirname(os.path.dirname(os.path.dirname(os.path.abspath(__file__))))
DATABASE_URL = os.getenv("DATABASE_URL", f"sqlite:///{_BASE_DIR}/lab2fhir.db")
```

### 1.2 文件上传缺乏安全校验，存在路径遍历和任意文件写入风险

- **文件**：`backend/src/api/routes_convert.py` 第30-32行
- **严重程度**：Critical
- **问题**：
  1. 直接使用 `file.filename` 拼接路径 `os.path.join(UPLOAD_DIR, file.filename)`，攻击者可以通过文件名 `../../etc/passwd` 实现路径遍历攻击。
  2. 没有任何文件大小限制，可能导致磁盘被大文件耗尽。
- **修复建议**：
  1. 使用 `os.path.basename()` 或 `secure_filename()` 过滤文件名中的路径分隔符。
  2. 生成 UUID 文件名，使用原始文件名仅作为元数据存储。
  3. 对上传文件添加大小限制。

```python
import uuid
safe_name = f"{uuid.uuid4().hex}.pdf"
file_path = os.path.join(UPLOAD_DIR, safe_name)
```

### 1.3 上传文件保存后无清理机制，存在磁盘泄漏

- **文件**：`backend/src/api/routes_convert.py` 第30-31行
- **严重程度**：Critical
- **问题**：上传的 PDF 文件保存到磁盘后，即使后续的文本提取或解析失败（第36-37行直接 return error），已保存的文件也不会被清理。长期积累会导致磁盘空间耗尽。
- **修复建议**：在转换流程中的所有错误路径上，使用 `try/finally` 确保临时文件被删除，或者仅在解析成功后才保留文件。

### 1.4 SQLite FTS5 索引存在并发写入风险

- **文件**：`backend/src/db/models.py` 第55-67行、第92-108行
- **严重程度**：Critical
- **问题**：
  1. `init_fts()` 和 `rebuild_fts()` 中通过 `sqlite3.connect(db_path)` 直接连接 SQLite，这与 SQLAlchemy 的 engine 使用不同的连接对象，绕过了 SQLAlchemy 的连接池管理。
  2. `rebuild_fts()` 函数中先创建新的 `SessionLocal()` 查询所有报告（第79-80行），然后直接对同一个数据库文件执行 `DELETE FROM reports_fts`（第77行），这两个操作使用不同连接，在并发场景下可能导致数据不一致。
  3. `_sync_fts` 监听器中 `try/except: pass`（第108行）会静默吞掉所有异常，FTS 同步失败时开发者完全无感知。
- **修复建议**：
  1. FTS5 操作应通过 SQLAlchemy 的 raw connection 执行，确保连接一致性。
  2. `rebuild_fts()` 应使用事务保护。
  3. `_sync_fts` 的异常至少应记录日志，不应完全吞掉。

### 1.5 报告详情端点存在 JSON 解析安全风险

- **文件**：`backend/src/api/routes_reports.py` 第89-117行
- **严重程度**：Critical
- **问题**：`get_report` 端点直接返回 `report.parsed_data` 和 `report.fhir_bundle`，这两个字段都是用户的输入数据。虽然 FastAPI 会对输出进行 JSON 序列化，但如果将来改为返回 HTML 或其他格式，存在 XSS 风险。更直接的问题是：`parsed_data` 和 `fhir_bundle` 可能包含原始 PDF 中的任何内容，没有任何输出过滤。
- **修复建议**：虽然没有即时的注入风险（因为是 JSON 输出），但建议在 API 响应 Schema 中明确定义结构，而不是透传原始 dict。

### 1.6 PDF 文件路径服务存在路径遍历风险

- **文件**：`backend/src/api/routes_reports.py` 第147-218行
- **严重程度**：Critical
- **问题**：
  1. `get_report_pdf` 中 `pdf_filename` 来自数据库，如果数据库中有恶意构造的文件名（如 `../../etc/shadow`），拼接 `os.path.join(PDF_TEST_DIR, report.pdf_filename)` 可能导致路径遍历。
  2. `serve_pdf_by_filename` 中 `filename:path` 捕获类型（第185行）接受任意 URL 路径，攻击者可以请求 `/api/pdf/../../etc/passwd` 读取系统文件。
  3. 第213行 `all(p in fname for p in parts if len(p) >= 4)` 的模糊匹配逻辑过于宽松，可能匹配到非预期的文件。
- **修复建议**：
  1. 对所有 pdf_filename 使用 `os.path.basename()` 过滤。
  2. 使用 `os.path.realpath()` 解析后，验证路径是否在 `PDF_TEST_DIR` 内部。
  3. 对于 `filename:path` 路由，限制匹配范围。

```python
safe_name = os.path.basename(report.pdf_filename)
exact = os.path.join(PDF_TEST_DIR, safe_name)
real = os.path.realpath(exact)
if not real.startswith(os.path.realpath(PDF_TEST_DIR)):
    raise HTTPException(403)
```

### 1.7 stats 缓存存在跨连接内存泄漏和并发竞态条件

- **文件**：`backend/src/db/models.py` 第112-163行
- **严重程度**：Critical
- **问题**：
  1. `_stats_cache` 是模块级全局变量，模块在进程生命周期内只会加载一次，缓存中的 `data` 对象会永久占用内存。
  2. `get_stats_cached()` 中 `recent` 列表（第137行）包含完整的 `Report` ORM 对象，这些对象关联了数据库连接 `session`。函数在 `db.close()` 后返回这些对象，调用者在访问这些对象的属性时可能触发 `DetachedInstanceError`。
  3. 多个请求同时调用 `get_stats_cached()` 且缓存过期时，会启动多个数据库查询，没有加锁保护。
- **修复建议**：
  1. 在 `recent` 中提前将 ORM 对象的属性提取为纯 Python 字典（代码中已部分实现但混用了 ORM 对象）。
  2. 添加简单的线程锁 `threading.Lock()` 防止重复查询。
  3. 使用 `functools.lru_cache` 或引入 proper caching 方案。

## 二、高危问题 (High)

### 2.1 数据库连接未在异常时正确关闭

- **文件**：`backend/src/db/repository.py` 第6-11行
- **严重程度**：High
- **问题**：`get_db()` 的 `yield` 模式在正常情况下工作正常，但 `SessionLocal()` 如果创建连接时抛出异常（如已超过最大连接数），`finally` 块中的 `db.close()` 不会被调用——因为 `db` 变量本身就是 `None`。虽然这在 SQLite 场景下几乎不会发生，但架构上不符合 FastAPI 的最佳实践。
- **修复建议**：异常处理应在 `try/finally` 块内。

### 2.2 报告去重检查使用裸查询，绕过了 Repository 模式

- **文件**：`backend/src/api/routes_convert.py` 第77-87行
- **严重程度**：High
- **问题**：去重逻辑直接使用 `db.query(Report).filter(...)` 裸查询，绕过 `ReportRepository` 层。这破坏了架构分层，也使得去重逻辑无法复用。同时 `created_at` 字段使用了 `server_default=func.now()`，但在 `create_report` 中重复提交时可能产生时间偏差。
- **修复建议**：将去重逻辑移到 `ReportRepository` 中，提供 `exists_by_pathology_id_and_type()` 方法。

### 2.3 文本解析正则表达式过于脆弱

- **文件**：`backend/src/parsers/patient_parser.py` 第17-19行、第32-35行等多处
- **严重程度**：High
- **问题**：多处正则使用 `(?:\s|$)` 来捕获字段值（如 `r'姓名[：:]\s*(.+?)(?:\s|$)'`），这会：
  1. 仅捕获到第一个空格为止，如果姓名、医院名等包含空格（如 "武警 上海市 总队医院"），只会抓到 "武警"。
  2. 报告文本中字段名和值可能跨行，当前正则完全不处理这种情况。
- **修复建议**：对于可能包含空格的值，使用更宽松的模式如 `r'姓名[：:]\s*(.+?)(?:\n|$)'` 或使用基于报告结构的上下文解析（如按行解析，查找字段名所在行的剩余部分）。

### 2.4 没有全局异常处理中间件

- **文件**：`backend/src/api/main.py` 第7-34行
- **严重程度**：High
- **问题**：FastAPI 应用没有注册任何异常处理中间件（exception handler）。当数据库查询失败、文件不存在、解析器崩溃等异常发生时，将直接返回 FastAPI 默认的 500 Internal Server Error，不会记录日志，也不会返回有意义的错误信息给前端。
- **修复建议**：添加全局异常处理器。

```python
from fastapi import Request
from fastapi.responses import JSONResponse

@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    # 记录日志
    return JSONResponse(status_code=500, content={"error": str(exc)})
```

### 2.5 CORS 配置过于宽松

- **文件**：`backend/src/api/main.py` 第13-19行
- **严重程度**：High
- **问题**：`allow_origins=["*"]` 允许任意来源访问 API，在生产环境中存在 CSRF 风险。虽然这是典型的开发配置，但应用没有区分开发/生产环境。
- **修复建议**：根据环境变量区分开发/生产配置，生产环境限制为具体的前端域名。

### 2.6 FTS 搜索无输入长度校验，可能触发数据库性能问题

- **文件**：`backend/src/db/repository.py` 第62-85行
- **严重程度**：High
- **问题**：
  1. `_fts_search` 接受任意长度输入，如果用户输入一个极长的搜索词，FTS5 可能触发 `LIKE` 替换逻辑造成单表全表扫描。
  2. SQLite FTS5 的 `MATCH` 查询中，如果特殊字符（如 `^`, `-`, `*` 等）没有被正确转义，会报错或不返回结果。
  3. 第72行 `replace('"', '').replace("'", "")` 仅是简单的引号去除，没有正确处理 FTS5 的所有特殊字符。
  4. 第78行 `LIMIT 500` 硬编码，超过 500 条匹配结果的搜索会丢失数据。
- **修复建议**：
  1. 限制搜索输入的最大长度（如 200 字符）。
  2. 使用 FTS5 的标准查询语法转义。
  3. `LIMIT` 改为与分页关联的动态值，或设置更高的上限。

### 2.7 Vue Router 缺少路由守卫

- **文件**：`frontend/src/router/index.js` 第1-41行
- **严重程度**：High
- **问题**：路由没有 navigation guards，任何路由可以直接通过 URL 访问。特别是对于 `/reports/:id`，如果 id 是无效的整数，详情页将直接显示空状态。对 `/patients/:name` 如果 name 不存在，也会显示空页面。
- **修复建议**：添加 `beforeEach` 守卫或在组件内验证参数有效性。

### 2.8 FTS5 rebuild 函数中的 SQL 注入风险

- **文件**：`backend/src/db/models.py` 第70-88行
- **严重程度**：High
- **问题**：`rebuild_fts()` 使用参数化查询插入 FTS 数据，这是正确的。但如果有其他代码直接使用字符串拼接构建 FTS 查询，将存在注入风险。当前 `_fts_search`（repository.py 第78行）也使用了参数化查询 `?`，这点是安全的。但 `_tokenize` 函数（第40-48行）仅在定义未使用，如果将来被用于非参数化查询，就是风险点。
- **修复建议**：统一说明使用参数化查询的规范，标记 `_tokenize` 函数为 deprecated 或移除。

## 三、中危问题 (Medium)

### 3.1 报告类型识别依赖硬编码的中文关键词

- **文件**：`backend/src/extractors/base.py` 第34-66行
- **严重程度**：Medium
- **问题**：`identify_report_type()` 完全依赖硬编码的中文关键词匹配。如果 PDF 文本因为 pdfplumber 提取的排版问题导致关键词断开（如 "DNA定量细胞学检查" 与 "报告" 被分在不同行），将无法识别。CLAUDE.md 中提到检测顺序很重要，但这个逻辑没有针对 "细胞学报告单" vs "TCT" 的可能边界情况做充分的测试覆盖。
- **修复建议**：考虑使用更鲁棒的规则，如基于关键词出现位置的加权评分。

### 3.2 PDF API 的 os.listdir() 每次请求都遍历文件系统

- **文件**：`backend/src/api/routes_reports.py` 第167-170行、第198-204行
- **严重程度**：Medium
- **问题**：每次请求 PDF 文件时，`os.listdir(PDF_TEST_DIR)` 都要遍历整个 405 个文件的目录。在高并发场景下，这会增加磁盘 I/O。
- **修复建议**：在应用启动时构建并缓存 `pathology_id -> filename` 的内存映射字典，定期或根据事件刷新。

### 3.3 前端 API 封装中静默吞掉错误

- **文件**：`frontend/src/api/index.js` 第11-87行
- **严重程度**：Medium
- **问题**：所有 API 函数都使用 `try/catch` 并返回默认值（如 `{ total: 0, ...}`、`null`、`[]`）。这种方式下：
  1. 调用者无法区分"数据为空的正常响应"和"网络错误"。
  2. 用户看到空白列表，不知道是暂无数据还是服务挂了。
  3. 错误信息完全丢失，不利于排查问题。
- **修复建议**：
  1. 保留原始错误信息，返回 `{ error: true, message: ... }` 结构。
  2. 或在全局级别使用 axios 拦截器统一处理错误，并在 UI 层展示错误状态。

### 3.4 前端多次使用相同的颜色映射逻辑（代码重复）

- **文件**：`frontend/src/views/ReportList.vue` 第68-74行、`frontend/src/views/ReportDetail.vue` 第33-35行、`frontend/src/views/PatientView.vue` 第40-42行、`frontend/src/views/ImportView.vue` 第186-188行
- **严重程度**：Medium
- **问题**：`report_type` 到 `el-tag` type 的映射逻辑至少在4个文件中重复出现，且不一致（ImportView 中的 tagType 映射缺少 `'细胞学'` 和 `'病理诊断'` 的类型值）。如果将来新增报告类型，需要修改多处代码。
- **修复建议**：提取为共用工具函数 `getReportTypeColor(type)` 放在 `utils/` 或 composable 中。

### 3.5 FHIR Bundle 生成器未处理嵌套结构的 parsed_data

- **文件**：`backend/src/fhir/generator.py` 第69-89行
- **严重程度**：Medium
- **问题**：Observation 序列化中，第72行的条件 `isinstance(value, (str, int, float))` 仅处理了标量类型。对于嵌套的 dict（如 DNA 分析表格、TCT 的细胞项目）和 list，不会生成对应的 Observation。虽然在临床实践中这些数据作为 codeable concept 更合适，但当前实现忽略了这部分结构化数据。
- **修复建议**：为 list 和 dict 类型添加序列化逻辑，或将复杂结构转化为包含多个 component 的单个 Observation。

### 3.6 Vue 组件数据初始化和 API 请求没有错误处理

- **文件**：`frontend/src/views/Dashboard.vue` 第97-116行、`frontend/src/views/ReportList.vue` 第128-151行、`frontend/src/views/ReportDetail.vue` 第200-204行、`frontend/src/views/PatientView.vue` 第85-89行
- **严重程度**：Medium
- **问题**：所有视图组件的 `onMounted` 中都没有 `try/catch` 错误处理。API 层虽然做了 try/catch，但视图层没有处理 API 返回空/异常的情况。特别地：
  - Dashboard：如果 `getStats()` 返回默认值 `{ total: 0, ... }`，视图会显示全零的统计卡片，用户无法判断是数据库为空还是服务不可用。
  - ReportDetail：`getReport()` 返回 `null` 时视图显示 "报告不存在"，但无法区分是网络错误还是真的不存在。
  - ReportList：搜索失败时静默显示空列表。
- **修复建议**：在视图层添加 loading/error/empty 三态管理，使用统一的错误提示组件。

### 3.7 批量导入可能因前一个文件失败而中断

- **文件**：`frontend/src/views/ImportView.vue` 第222-251行
- **严重程度**：Medium
- **问题**：批量导入中如果某个文件的 `uploadPdf()` 因为网络问题抛出未捕获异常（非 reject 的 promise），整个 `for` 循环会中断，其余文件不会被处理。
- **修复建议**：对每个 `uploadPdf()` 调用添加独立的 try/catch 包裹，确保单个失败不影响其余文件。

### 3.8 PDF 查看器无加载超时处理

- **文件**：`frontend/src/components/PdfViewer.vue` 第59-75行
- **严重程度**：Medium
- **问题**：PDF 文件加载失败或网络超时时，`loadPdf()` 会持续停留在 loading 状态。没有超时机制，用户会在 loading 状态永远等待。
- **修复建议**：使用 `Promise.race` 添加超时（如30秒），超时后显示错误提示并提供重试按钮。

### 3.9 无日志记录机制

- **文件**：`backend/src/api/main.py`、所有后端文件
- **严重程度**：Medium
- **问题**：整个后端没有任何日志记录框架（如 Python logging）。PDF 提取错误仅在 `extractors/base.py` 第19行使用 `print()` 输出。没有请求日志、错误日志、性能日志，生产环境排查问题极其困难。
- **修复建议**：集成 Python `logging` 模块，至少记录请求路径、状态码、解析错误和数据库异常。

### 3.10 index.html 语言不匹配

- **文件**：`frontend/index.html` 第2行
- **严重程度**：Medium
- **问题**：`<html lang="en">` 声明页面语言为英语，但整个应用是中文界面。这会影响屏幕阅读器的发音和浏览器的自动翻译行为。
- **修复建议**：改为 `<html lang="zh-CN">`。

### 3.11 单文件上传 API 在批量上传中被逐个顺序调用

- **文件**：`frontend/src/views/ImportView.vue` 第222-251行（使用 uploadPdf）和 `frontend/src/api/index.js` 第56-69行
- **严重程度**：Medium
- **问题**：
  1. 前端的批量上传逐文件调用 `/api/convert`（每个文件一次 HTTP 请求），而不是使用后端提供的 `/api/convert/batch` 接口。后端 batch 接口接收多文件但实际实现也是逐文件循环调用（`routes_convert.py` 第124-128行）。
  2. 前后端都没有做并发控制，无法充分利用并发上传。
- **修复建议**：
  1. 后端 batch 接口改为使用 `asyncio.gather` 并行处理。
  2. 前端使用后端 batch 接口（仅一次 HTTP 请求），或保留逐个上传但添加并发限制（如最多3个并发）。

## 四、低危问题 (Low)

### 4.1 代码中残留注释掉的函数调用

- **文件**：`backend/src/api/main.py` 第29行
- **严重程度**：Low
- **问题**：`# rebuild_fts()  # 仅在需要重建索引时手动调用` 建议将此功能做成独立的 CLI 命令或管理 API，而不是注释掉的调用。
- **修复建议**：移除注释或创建独立的管理命令。

### 4.2 使用已弃用的 FastAPI on_event 装饰器

- **文件**：`backend/src/api/main.py` 第25行
- **严重程度**：Low
- **问题**：`@app.on_event("startup")` 在 FastAPI 0.93+ 中已被标记为 deprecated，推荐使用 `lifespan` context manager。
- **修复建议**：迁移到 `@asynccontextmanager lifespan` 模式。

### 4.3 重复定义的 label 映射

- **文件**：`backend/src/db/models.py` 第117-119行和 `backend/src/db/repository.py` 第108-115行
- **严重程度**：Low
- **问题**：`_TYPE_LABELS`（models.py）和 `_type_label` 静态方法（repository.py）定义了完全相同的报告类型标签映射。FHIR generator 中 `_report_type_label` 又是第三份重复定义。违反 DRY 原则。
- **修复建议**：统一为单一数据源，或放在常量文件中。

### 4.4 FHIR Bundle 的时间戳使用字符串而非标准格式

- **文件**：`backend/src/fhir/generator.py` 第94行
- **严重程度**：Low
- **问题**：`"timestamp": str(date.today())` 生成的是 `"2026-07-13"` 格式，但 FHIR R4 标准中 `Bundle.timestamp` 应为 ISO 8601 格式的 `instant` 类型（含时间及时区，如 `"2026-07-13T00:00:00+08:00"`）。
- **修复建议**：使用 `datetime.now().isoformat()`。

### 4.5 DNA 解析中数字解析无异常保护

- **文件**：`backend/src/parsers/dna_parser.py` 第23-40行
- **严重程度**：Low
- **问题**：`float(m.group(1))` 和 `int(m.group(1))` 在文本格式异常时会抛出 `ValueError`。虽然有正则做前置过滤，但在边界情况下（如数值后有多余字符，被上一行的正则捕获到），仍可能崩溃。
- **修复建议**：添加 `try/except ValueError` 的回退处理。

### 4.6 API 返回 `return {"error": "not found"}, 404` 不符合 FastAPI 惯例

- **文件**：`backend/src/api/routes_reports.py` 第94行
- **严重程度**：Low
- **问题**：FastAPI 中抛出 404 的标准方式是 `raise HTTPException(status_code=404, detail="...")`。手动 `return dict, 404` 虽然能工作，但不会触发 OpenAPI 文档中正确的错误响应声明。
- **修复建议**：改为 `raise HTTPException(status_code=404, detail="报告不存在")`。

### 4.7 缺少 favicon.svg 文件引用但文件可能不存在

- **文件**：`frontend/index.html` 第5行
- **严重程度**：Low
- **问题**：引用了 `/favicon.svg` 但项目中没有看到这个文件，浏览器控制台会产生 404 错误。
- **修复建议**：添加 favicon.svg 文件或使用内联 SVG/data URI。

### 4.8 所有 API 使用同步 def（无 async/await 收益）

- **文件**：`backend/src/api/routes_reports.py` 各处
- **严重程度**：Low
- **问题**：所有路由函数使用同步 `def` 声明，而 SQLAlchemy 的同步 Session 操作会阻塞事件循环。虽然 FastAPI 会把同步函数放到线程池执行，但使用 `async def` + `async for` 配合 SQLAlchemy 异步模式会有更好的并发性能。
- **修复建议**：在当前 SQLite 场景下影响不大，但如果将来迁移到 PostgreSQL，建议使用 async SQLAlchemy。

### 4.9 Dashboard.vue 中使用内联样式过多

- **文件**：`frontend/src/views/Dashboard.vue` 第17-56行
- **严重程度**：Low
- **问题**：大量内联样式与 scoped/global CSS 混用，降低可维护性。特别的分隔线、flex 布局等应提取为 CSS 类。
- **修复建议**：将可复用的布局样式提取到 `global.css` 或组件级 `<style scoped>` 中。

### 4.10 HPV 分型提取可能漏掉一些命名变体

- **文件**：`backend/src/parsers/hpv_parser.py` 第40-46行
- **严重程度**：Low
- **问题**：`r'HPV(\d+)'` 匹配具体分型，但某些报告可能写为 "HPV-16" 或 "HPV 16"（带连字符或空格），可能漏掉。
- **修复建议**：改为 `r'HPV[- ]?(\d+)'`。

### 4.11 gastric_parser 的诊断过滤关键词列表不完整

- **文件**：`backend/src/parsers/gastric_parser.py` 第41-43行、`backend/src/parsers/pathology_parser.py` 第67-70行
- **严重程度**：Low
- **问题**：用于排除非诊断内容的 `skip_kw` 列表在两个文件中各自维护，且不完全一致。随着更多报告变体出现，维护成本增加。
- **修复建议**：合并为 `common_exclusion_keywords` 元组并共享引用。

## 五、性能问题 (Performance)

### 5.1 stats 缓存未考虑缓存命中率

- **文件**：`backend/src/db/models.py` 第122-163行
- **严重程度**：Medium
- **问题**：`stats_cache` 在数据变更时（`create_report` 中调用 `invalidate_stats_cache()`）立即失效。如果导入 405 份报告，每次 `create_report` 都会触发缓存失效，下一次请求会重新计算全量统计——这发生在紧接的多次请求中。
- **修复建议**：使用带短 TTL（如 10 秒）而非即时失效的策略，或批量导入后统一失效。

### 5.2 大 JSON 字段每次请求都完整传输

- **文件**：`backend/src/api/routes_reports.py` 第89-117行
- **严重程度**：Medium
- **问题**：`get_report` 返回完整的 `parsed_data` 和 `fhir_bundle` JSON。这两个字段可能非常大（特别是 FHIR Bundle），在报告详情页中每次用户刷新都会完整传输。`list_reports` 中已正确排除这些字段。
- **修复建议**：对于详情页，考虑支持增量加载或分块传输。至少在前端缓存已经加载过的报告数据。

### 5.3 PDF.js worker 从 CDN 加载

- **文件**：`frontend/src/components/PdfViewer.vue` 第40行
- **严重程度**：Medium
- **问题**：`pdf.worker.min.mjs` 每次从 CloudFlare CDN 加载，增加了外部依赖和延迟。在内网部署场景中完全无法工作。
- **修复建议**：将 pdfjs-dist 的 worker 文件包含在项目中，由 Vite 构建时打包。

## 六、代码质量问题 (Code Quality)

### 6.1 API 路由序列化逻辑重复

- **文件**：`backend/src/api/routes_reports.py` 多处
- **严重程度**：Medium
- **问题**：报告对象的序列化逻辑在 `list_reports`（第60-79行）、`get_report`（第96-117行）和 `get_patient_reports`（第124-143行）中基本重复三次，每次仅增减几个字段。任何字段变更需要修改三处。
- **修复建议**：提取为独立的 `_serialize_report(report: Report, full: bool = False)` 函数。

### 6.2 缺少类型注解

- **文件**：大多数后端文件
- **严重程度**：Low
- **问题**：函数参数和返回值大量缺少类型注解（PEP 484）。虽然 Python 3.10+ 支持 `str | None` 语法（已在部分地方使用），但很多函数（如所有 parser 函数）没有完整的参数和返回值类型声明。
- **修复建议**：为所有函数添加类型注解，利用 mypy 或 pyright 进行静态检查。

### 6.3 报告类型字符串魔法值散布

- **文件**：所有后端和前端文件
- **严重程度**：Low
- **问题**：报告类型字符串 `"DNA"`, `"TCT"`, `"HPV"`, `"胃镜"`, `"细胞学"`, `"病理诊断"` 在代码中作为硬编码字符串散布在数十处。修改任一类型名需要全量搜索替换。
- **修复建议**：定义枚举 `ReportType(str, Enum)` 并在全局使用。

## 七、架构问题 (Architecture)

### 7.1 解析器与 API 路由紧密耦合

- **文件**：`backend/src/api/routes_convert.py` 第48-59行
- **严重程度**：Medium
- **问题**：报告类型与解析器的映射通过 if/elif 链硬编码在路由中。新增报告类型需要同时修改 parser 包和 API 路由。违反了开闭原则 (Open/Closed Principle)。
- **修复建议**：使用注册表模式或工厂模式动态发现解析器。

```python
PARSER_REGISTRY = {
    "DNA": parse_dna,
    "TCT": parse_tct,
    "HPV": parse_hpv,
    "胃镜": parse_gastric,
    "细胞学": parse_cytology,
    "病理诊断": parse_pathology,
}
parsed_data, diagnosis = PARSER_REGISTRY.get(report_type, lambda x: ({}, ""))(raw_text)
```

### 7.2 缺少数据库迁移机制

- **文件**：`backend/src/db/models.py` 第51-52行
- **严重程度**：Medium
- **问题**：使用 `Base.metadata.create_all()` 创建数据库表，但没有任何迁移管理工具（如 Alembic）。如果模型变更（如新增字段、修改字段类型），已有的数据库不会自动升级，只能手动操作或删除重建。
- **修复建议**：集成 Alembic 数据库迁移管理。

### 7.3 前端缺少全局状态管理

- **文件**：所有前端视图文件
- **严重程度**：Low
- **问题**：当前没有使用 Pinia 或 Vuex 进行全局状态管理。dashboard、report list、patient view 各自独立调用 API 获取数据，没有数据共享或缓存机制。用户从列表页进入详情页再返回，列表会重新请求数据。
- **修复建议**：对于搜索/筛选状态和报告列表缓存，考虑引入 Pinia store。使用 `keep-alive` 对列表组件缓存，避免不必要的重新请求。

### 7.4 后端缺少配置管理

- **文件**：`backend/src/db/models.py` 第5行、`backend/src/api/routes_convert.py` 第19行
- **严重程度**：Low
- **问题**：配置值（数据库路径、上传目录、PDF测试目录）散布在多个文件中，通过硬编码路径和 `os.getenv` 各自管理。没有统一的配置类。
- **修复建议**：使用 Pydantic Settings（`pydantic-settings`）创建集中化的配置类，支持从环境变量和 `.env` 文件加载。
